## Commands

All commands run from `extension/` (the only component with tooling so far; `backend/` has none yet). Example: `npm --prefix extension run build`.

- `npm install` ‚Äî install dependencies (first-time setup).
- `npm run dev` ‚Äî Vite dev server with CRXJS hot-reload, for live development against Gmail.
- `npm run build` ‚Äî typecheck (`tsc --noEmit`) then production build to `extension/dist/` (the loadable unpacked extension).
- `npm run typecheck` ‚Äî type-check only, no build.
- `npm run lint` ‚Äî ESLint (typescript-eslint + react-hooks + react-refresh; Prettier-compatible).
- `npm run format` / `npm run format:check` ‚Äî Prettier (default config) write / check.
- `npm run test` ‚Äî Vitest (jsdom + React Testing Library), single run.
- `npm run icons` ‚Äî regenerate the throwaway placeholder dev icons (real brand work is in `PRE_LAUNCH_CHECKLIST.md`).

**Load the unpacked extension:** Chrome ‚Üí `chrome://extensions` ‚Üí enable Developer mode ‚Üí "Load unpacked" ‚Üí select `extension/dist/`.

**Non-obvious gotchas (encoded here so future sessions don't rediscover them):**
- `npm audit` flags 2 high-severity advisories in **dev-only** transitive deps (`rollup` via `@crxjs/vite-plugin`). `npm audit --omit=dev` is clean ‚Äî nothing vulnerable ships in the extension. Do **not** run `npm audit fix --force` (it breaks the build). This is the accepted CRXJS community-plugin tax (a locked, eyes-open trade-off ‚Äî see Locked tech decisions).
- The build prints a benign CRXJS-on-Vite-8 warning ("Both `rollupOptions` and `rolldownOptions` ‚Ä¶ will be ignored"). Content scripts still build and wire up correctly; it is cosmetic and internal to CRXJS.
- `vitest.config.ts` is intentionally excluded from `tsconfig.json`'s `include` (a Vite 8 ‚Üî Vitest-nested-Vite types skew, not a code defect). Test code under `src/` is still fully type-checked.
- **MV3 entry files must have distinct basenames.** Under Vite 8/Rolldown, CRXJS collapses two entries that share a basename (e.g. `src/background/index.ts` + `src/content/index.ts`) into one chunk, and the service worker silently ends up running the content script's code (no error; onboarding just never opens). The fix already applied: the entries are `src/background/service-worker.ts` and `src/content/content-script.ts`. Do not rename them back to `index.ts`. To sanity-check after a build: `dist/service-worker-loader.js` must import the `service-worker.ts-*` chunk, not the `content-script.ts-*` one.
- **Local storage carries a `schemaVersion`.** The `outboxiqState` root (PRD ¬ß7.2) has a `schemaVersion` field (currently `1`, the `SCHEMA_VERSION` constant in `src/lib/constants.ts`). There is intentionally **no migration framework yet** ‚Äî only the field. When a future change alters the ¬ß7.2 shape, bump `SCHEMA_VERSION` and branch on the stored value in `getState()` to migrate old records.
- **All Gmail DOM automation is isolated to one module.** `src/lib/schedule/gmail-recipe.ts` is the *single point of failure* for Gmail UI changes ‚Äî every selector and the synthetic-event recipe live there and nowhere else. Its header points to the canonical references: `research/scheduled-send-api-spike.md` (Verification section, preset path), `research/pick-date-time-probe.md` (custom path + relabel-target dump), and `research/send-button-probe.md` (the ¬ß5.5.1 regular-Send button + the whole-gesture / capture-phase / replay facts). Run the matching probe to rediscover selectors when Gmail breaks: Schedule Send ‚Üí the spike/pick-date-time probes; the regular-Send guard ‚Üí `send-button-probe.js` (`discover()` ‚Üí `watch()` ‚Üí `armBlockGesture()` ‚Üí `testReplay()`). When either breaks post-launch, fix it here.
- **¬ß5.5.1 intercepts the WHOLE Send gesture, not one event.** The probe proved Gmail finalises the send on a *later* event of the gesture (pointerup/click), so the regular-Send guard captures pointerdown+mousedown+pointerup+mouseup+click **and** ‚åò/Ctrl+Enter ‚Äî a single-event hook leaks (the one-shot `armSuppress` "sent anyway" that the gate caught). It only ever `preventDefault`s on a *certain* violation (decided synchronously off the cached config); everything else falls through to native Send. Do not "simplify" it to a click-only listener.
- **React inside Shadow DOM (the ¬ß5.3 modal).** Quirks, encoded so they aren't rediscovered: (1) `createRoot` must target an element **inside** the shadow root, not the host ‚Äî since React 17 event delegation roots at the `createRoot` container, so rooting it in the shadow tree is what makes synthetic events fire for shadow-rendered controls; (2) page CSS does not cross the shadow boundary, so the modal's styles **must** be injected into the shadow root (`schedule-modal/styles.ts`) ‚Äî the upside is hard isolation in both directions (Gmail can't break us, we can't leak into Gmail); (3) keep one host at a time ‚Äî re-opening removes the previous host so a stale React root can't linger. See `src/content/schedule-modal/mount.tsx`.
- **The content script now bundles React** (~190 kB `jsx-runtime` chunk) because the ¬ß5.3 modal is React and is imported by `content-script.ts`. It loads on every Gmail page. Acceptable for v1/MVP; a later optimisation is to lazy-load the modal bundle only when the user actually triggers Schedule Send. Flagged, not yet done.

## Architecture

Two independently-deployed components, kept deliberately small:

### `extension/` ‚Äî Chrome MV3 (does almost everything)

Local-first by design. All recipient-timezone detection, scheduling UI, working-hours logic, and storage runs in the browser. The extension **enhances** Gmail's native Schedule Send rather than replacing it ‚Äî the native dropdown is rebranded and the native modal is enriched, but Gmail's interaction patterns and the Scheduled label are preserved (PRD ¬ß8.1 "native feel over branded feel"). Planned layout: `src/background/` (service worker, OAuth, backend messaging), `src/content/` (Gmail DOM injection), `src/pages/{onboarding,settings}/`, `src/lib/` (shared utilities), `public/icons/`.

### `backend/` ‚Äî EU-hosted relay (exists for exactly one reason)

The backend is **not** a general server. It exists for one specific server-side concern and nothing else:

1. **Unschedule-on-Reply relay** ‚Äî subscribes to Gmail push notifications via Google Cloud Pub/Sub and cancels scheduled emails when a reply arrives (PRD ¬ß5.6, ¬ß7.3).

> **Google Maps APIs were removed from product scope (2026-05-16, owner-directed ‚Äî not deferred, removed).** The former second purpose (Maps Geocoding/Time Zone proxy for recipient-timezone resolution) is gone entirely: no Maps API key, billing, OAuth scope, or proxy endpoint, ever. Recipient timezone resolution is fully on-device (cache ‚Üí People API ‚Üí Workspace Directory ‚Üí manual, cached forever). Rationale + counterfactual: PRD ¬ß5.4.1 amendment, ¬ß7.3.1, `notes/owner-decisions-log.md` Entry 26. Do not reintroduce a Maps dependency without re-opening that decision.

Do not add other endpoints, telemetry sinks, user-data storage, or "while we're here" features. If you find yourself wanting to, that's a signal to push the logic back into the extension or to stop.

**Data rules:** no email content is ever stored. Stored data is limited to user email, encrypted OAuth refresh token (per-user keys), and active scheduled-message records. EU region (Frankfurt/Amsterdam) for GDPR data-residency.

## Locked product decisions

Product/UX decisions locked by the owner ‚Äî do not re-litigate without explicit user input:

- **¬ß5.5 working-hours / absolute-limits enforcement is a soft warning, not a hard block and not a silent auto-snap (locked 2026-05-16, Session 5).** When a user tries to schedule (or send) an email at a time that violates their configured working hours or absolute limits, show a soft-warning modal that names the violation in plain language (e.g. "You set 7:00 AM as your earliest send time. This email is scheduled for 3:33 AM. What would you like to do?") and offers exactly three explicit choices: **(1) Proceed anyway** (honour the user's specified time, overriding the limit just this once); **(2) Snap to [the violated boundary]** (e.g. 7:00 AM, or the relevant working-hours/absolute boundary for the rule that was violated); **(3) Cancel** (close the modal, leave the user in compose to adjust). Rationale: a hard block is too paternalistic (legitimate overrides exist, e.g. a client in another timezone) and damages trust; a silent auto-snap is surprising and violates ¬ß8.3 "sensible defaults, easy overrides" (the user must always know what's happening to their email); a soft warning gives agency ‚Äî it acknowledges the violation, surfaces options, and requires an explicit choice, matching the PRD's own ¬ß5.5 modal pattern. The **same pattern applies to both** working-hours violations and absolute-limits violations. If **both are violated at once, surface the absolute-limit violation** (it is the harder of the two constraints). Do not replace this with a block, an auto-snap, or a 2-option variant in any future session.

- **Warnings fire for unintended actions, not the product's core use case (locked 2026-05-16, Session 5.5).** The ¬ß5.5 trigger is **split**: **Schedule Send warning = absolute limits only**; **Regular Send (¬ß5.5.1) warning = working hours + absolute limits**. Reasoning: deliberately *scheduling* outside one's working hours to land in a recipient's window is exactly what OutboxIQ is *for* ‚Äî warning on it trains users to dismiss the modal and guts its value for the absolute-limit case where it matters. An *immediate* off-hours Send, by contrast, is plausibly unintended, so ¬ß5.5.1 warns on both. The ¬ß5.5 calc (`working-hours.ts`) still computes **both** rule types unconditionally ‚Äî **do not delete the working-hours branch**; it is consumed by ¬ß5.5.1 (Session 5.6) and ¬ß5.3.5 recipient-optimization (Session 6, advisory input only). The **default 9 AM working-hours start is retained**: under this split it does *not* nag Schedule Send, but it *does* drive the ¬ß5.5.1 regular-Send prompt (clicking Send at 8:30 AM ‚Üí "you're sending before your stated start time ‚Äî schedule for 9 AM instead?"). That prompt is **signal, not noise** ‚Äî a user whose real pattern differs adjusts their working hours in onboarding. Do not re-litigate the default or the split.

- **¬ß5.5 snap-target resolution (locked 2026-05-16, Session 5).** "Snap to [boundary]" resolves differently per rule type. **Working-hours violation** ‚Üí the ¬ß5.5.3 next-working-day algorithm (next configured working day at that day's working-hours start). **Absolute-limit violation** ‚Üí the absolute boundary (`absoluteEarliest`/`absoluteLatest`) **on the same calendar day the user originally picked** ‚Äî *not* next-working-day. Reasoning: absolute limits are *clock-time* rules ("never send before 7 AM / after 7 PM in my local time"), not day-of-week rules; the user chose their target day deliberately when setting the custom time, so the snap fixes only the clock-time violation while honouring the day-of-week intent. Consistent with PRD ¬ß5.1.3's "Earliest I'd ever send an email" framing.
- **¬ß5.5 absolute snap may land on a non-working day ‚Äî intentionally permitted in v1 (locked 2026-05-16, Session 5).** Because an absolute snap keeps the user's chosen day (above), it can land on a non-working day (e.g. Sat 7:00 AM with Mon‚ÄìFri working hours). This is **deliberate**: the user's explicit day choice overrides the working-hours *soft* preference (working hours are a soft preference; absolute limits are the hard rule being enforced here). Do **not** "fix" this in v1 by re-snapping to a working day or chaining a second warning. A v2 enhancement *could* surface a secondary informational note in the modal ("this lands outside your working hours, but within your absolute limits") ‚Äî **deferred, not v1 scope; do not implement.**
- **¬ß5.5.1 regular-Send-button trigger ‚Äî deferred (Session 5 owner decision).** ¬ß5.5 enforcement this session guards only times the user explicitly picks via OutboxIQ's Schedule Send modal. Intercepting Gmail's *primary Send* button when the current time is outside working hours (PRD ¬ß5.5.1 first bullet) is its own future session ‚Äî far more invasive and a direct ¬ß5.2.3 ("never block native Gmail") risk. The PRD ¬ß5.5.1 first bullet remains spec; it is simply not yet implemented.
- **¬ß5.5.2 auto-suppress footer checkbox ‚Äî dropped for v1 (Session 5 owner decision).** The locked soft-warning pattern requires an explicit choice every time, so the PRD ¬ß5.5.2 "Always schedule for next working day when outside hours" suppression checkbox is not built; PRD ¬ß5.5.2 is amended accordingly. Consequence to note (and not "fix"): the `alwaysScheduleOutsideHours` field in the ¬ß7.2 storage schema and its planned ¬ß5.8 Settings toggle are **inert in v1** by design ‚Äî left in place (schema stability) but unused. Do not wire them to silent auto-snap behaviour; that is exactly the "surprising silent shift" the locked decision rejects.
