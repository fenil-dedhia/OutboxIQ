**Non-obvious gotchas (encoded here so future sessions don't rediscover them):**
- **`npm run dev` corrupted `dist/` (Session 8).** A `npm run dev` run left `extension/dist/` containing `node_modules`, the `extension-key.pem`, the full source tree, and a recursively-nested `dist/dist/`, and the service worker would not open from `chrome://extensions`. A clean one-shot `npm run build` / `build:smoke` (with `publicDir:false`) does **not** do this — root cause in CRXJS dev mode not chased (the hands-on smoke path was moved off the dev server to `build:smoke` instead — Entry-25 "design for the real constraint"). Recovery: `rm -rf extension/dist` then a one-shot build; in Chrome **Remove** the old card and **Load unpacked** the fresh `dist/`. Prefer one-shot builds for hands-on verification; treat `npm run dev` as suspect until its dist output is re-validated.
- **Free v1 must NOT import anything from `src/premium-v1/` (Entry 39).** The service worker deliberately does **not** `import "./oauth"` — Free v1 has no OAuth. The OAuth/People stack lives inert in `src/premium-v1/`; wiring it into any Free v1 entry (`service-worker.ts`, `content-script.ts`, pages) re-introduces the install-time `identity`/host permissions and the CASA/consent-verification gate the Entry-39 pivot removed. Audit before shipping: `grep -rn 'from "[^"]*premium-v1' src --include=*.ts | grep -v src/premium-v1/` must be empty. (Premium v1 wires it up later — `src/premium-v1/README.md`.)
- `npm audit` flags 2 high-severity advisories in **dev-only** transitive deps (`rollup` via `@crxjs/vite-plugin`). `npm audit --omit=dev` is clean — nothing vulnerable ships in the extension. Do **not** run `npm audit fix --force` (it breaks the build). This is the accepted CRXJS community-plugin tax (a locked, eyes-open trade-off — see Locked tech decisions).
- The build prints a benign CRXJS-on-Vite-8 warning ("Both `rollupOptions` and `rolldownOptions` … will be ignored"). Content scripts still build and wire up correctly; it is cosmetic and internal to CRXJS.
- `vitest.config.ts` is intentionally excluded from `tsconfig.json`'s `include` (a Vite 8 ↔ Vitest-nested-Vite types skew, not a code defect). Test code under `src/` is still fully type-checked.
- **MV3 entry files must have distinct basenames.** Under Vite 8/Rolldown, CRXJS collapses two entries that share a basename (e.g. `src/background/index.ts` + `src/content/index.ts`) into one chunk, and the service worker silently ends up running the content script's code (no error; onboarding just never opens). The fix already applied: the entries are `src/background/service-worker.ts` and `src/content/content-script.ts`. Do not rename them back to `index.ts`. To sanity-check after a build: `dist/service-worker-loader.js` must import the `service-worker.ts-*` chunk, not the `content-script.ts-*` one.
- **Local storage carries a `schemaVersion`.** The `outboxiqState` root (PRD §7.2) has a `schemaVersion` field (currently `3`, the `SCHEMA_VERSION` constant in `src/lib/constants.ts`). There is intentionally **no migration framework yet** — only the field. So far every bump has been *purely additive*, so `getState()`'s default-merge IS the migration (no version branch needed yet): v1→v2 added `lastScheduled`, v2→v3 added `pinnedTimezones` (Session 11, defaults to `[]`). When a future change alters the §7.2 shape *non-additively*, bump `SCHEMA_VERSION` and branch on the stored value in `getState()`.
- **Gmail DOM knowledge is isolated to two modules, each with a probe.** (1) `src/lib/schedule/gmail-recipe.ts` — the *single point of failure* for Gmail Schedule-Send automation (every selector + the synthetic-event recipe). Canonical refs: `research/scheduled-send-api-spike.md` (preset path), `research/pick-date-time-probe.md` (custom path + relabel target), `research/send-button-probe.md` (§5.5.1 Send button + whole-gesture/capture/replay). (2) `src/content/compose/compose-recipients.ts` — the single point of failure for **§5.3.5 compose-recipient reading**; anchor `div[role="option"][data-hovercard-id]` scoped to the chevron's compose pane (canonical ref: `research/compose-recipients-probe.{js,md}`, live-verified 2026-05-20 — the originally-shipped `input[name="to"|"cc"]`/`[email]` guess was wrong). Run the matching probe to rediscover selectors when Gmail breaks: Schedule Send → spike/pick-date-time probes; regular-Send guard → `send-button-probe.js`; recipient reading → `compose-recipients-probe.js` (`__flComposeProbe.discover()`). When any breaks post-launch, fix it in that one module.
- **§5.5.1 intercepts the WHOLE Send gesture, not one event.** The probe proved Gmail finalises the send on a *later* event of the gesture (pointerup/click), so the regular-Send guard captures pointerdown+mousedown+pointerup+mouseup+click **and** ⌘/Ctrl+Enter — a single-event hook leaks (the one-shot `armSuppress` "sent anyway" that the gate caught). It only ever `preventDefault`s on a *certain* violation (decided synchronously off the cached config); everything else falls through to native Send. Do not "simplify" it to a click-only listener.
- **React inside Shadow DOM (the §5.3 modal).** Quirks, encoded so they aren't rediscovered: (1) `createRoot` must target an element **inside** the shadow root, not the host — since React 17 event delegation roots at the `createRoot` container, so rooting it in the shadow tree is what makes synthetic events fire for shadow-rendered controls; (2) page CSS does not cross the shadow boundary, so the modal's styles **must** be injected into the shadow root (`schedule-modal/styles.ts`) — the upside is hard isolation in both directions (Gmail can't break us, we can't leak into Gmail); (3) keep one host at a time — re-opening removes the previous host so a stale React root can't linger. See `src/content/schedule-modal/mount.tsx`.
- **The content script now bundles React** (~190 kB `jsx-runtime` chunk) because the §5.3 modal is React and is imported by `content-script.ts`. It loads on every Gmail page. Acceptable for v1/MVP; a later optimisation is to lazy-load the modal bundle only when the user actually triggers Schedule Send. Flagged, not yet done.

## Architecture

Two independently-deployed components, kept deliberately small:

### `extension/` — Chrome MV3 (does almost everything)

Local-first by design. All recipient-timezone detection, scheduling UI, working-hours logic, and storage runs in the browser. The extension **enhances** Gmail's native Schedule Send rather than replacing it — the native dropdown is rebranded and the native modal is enriched, but Gmail's interaction patterns and the Scheduled label are preserved (PRD §8.1 "native feel over branded feel"). Layout: `src/background/` (Free v1 service worker — **no OAuth**; cache→manual tz resolver), `src/content/` (Gmail DOM injection), `src/pages/{onboarding,settings}/`, `src/lib/` (shared utilities), `src/premium-v1/` (**inert, preserved Premium OAuth/People stack — not imported by Free v1**; see its README), `public/icons/`. **Free v1 makes no Google API calls (Entry 39).**

### `backend/` — EU-hosted relay (Premium v1 only; exists for exactly one reason)

> **Premium v1 scope (Entry 32).** Not part of Free v1 — Free v1 ships extension-only. This block describes the **Premium v1** backend (PRD §13), gated by `PREMIUM_LAUNCH_CHECKLIST.md`. It is **not the next work**.

The backend is **not** a general server. It exists for one server-side concern and nothing else: the **Unschedule-on-Reply relay** — subscribes to Gmail push notifications via Google Cloud Pub/Sub and cancels scheduled emails when a reply arrives (PRD §5.6, §7.3).

- **Google Maps was removed from product scope (Entry 26 — removed, not deferred).** The former Maps Geocoding/Time Zone proxy is gone entirely: no Maps API key, billing, OAuth scope, or proxy endpoint, ever. Recipient timezone is fully on-device (cache → People → Workspace Directory → manual). Do not reintroduce without re-opening Entry 26. Rationale: PRD §5.4.1 amendment, §7.3.1.
- **OAuth token management is part of the one purpose, not a second one (Entry 31; PRD §7.5/§7.3.1).** The backend's purpose is Unschedule-on-Reply *and the OAuth token management that enables it*: §5.6 needs backend-initiated Gmail calls when the extension isn't running ⇒ a backend refresh token ⇒ the backend runs the auth-code exchange ("Option B"), necessarily reused by the extension's other Google calls (one grant per user). Not a second purpose; not a reversal of Entry 26.

Do not add other endpoints, telemetry sinks, user-data storage, or "while we're here" features. If you find yourself wanting to, that's a signal to push the logic back into the extension or to stop.

**Data rules:** no email content is ever stored. Stored data is limited to user email, encrypted OAuth refresh token (per-user keys), and active scheduled-message records. EU region (Frankfurt/Amsterdam) for GDPR data-residency.

## Locked product decisions

Product/UX decisions locked by the owner — do not re-litigate without explicit user input. (Reasoning + counterfactuals in the cited entries.)

- **§5.5 enforcement is a soft warning, not a hard block and not a silent auto-snap (locked 2026-05-16; Entries 19–20).** On a working-hours or absolute-limits violation, show a soft-warning modal that names the violation in plain language (e.g. "You set 7:00 AM as your earliest send time. This email is scheduled for 3:33 AM. What would you like to do?") and offers exactly three explicit choices: **(1) Proceed anyway** (honour the user's time, overriding just this once); **(2) Snap to [the violated boundary]**; **(3) Cancel** (close, leave the user in compose to adjust). A hard block is too paternalistic (legitimate overrides exist) and a silent auto-snap violates §8.3 "sensible defaults, easy overrides" — a soft warning gives agency. Same pattern for both rule types; **if both are violated at once, surface the absolute-limit violation** (the harder constraint). Do not replace this with a block, an auto-snap, or a 2-option variant.
- **Warnings fire for unintended actions, not the product's core use case (locked 2026-05-16; Entry 21).** The §5.5 trigger is **split**: **Schedule Send warning = absolute limits only**; **Regular Send (§5.5.1) warning = working hours + absolute limits**. Deliberately *scheduling* outside one's working hours to land in a recipient's window is exactly what Fashionably Late is *for* — warning on it trains users to dismiss the modal. An *immediate* off-hours Send is plausibly unintended, so §5.5.1 warns on both. The calc (`working-hours.ts`) still computes **both** rule types unconditionally — **do not delete the working-hours branch**; it is consumed by §5.5.1 and §5.3.5 (advisory input). The **default 9 AM working-hours start is retained**: under this split it does not nag Schedule Send but does drive the §5.5.1 regular-Send prompt — signal, not noise (a user whose pattern differs adjusts onboarding). Do not re-litigate the default or the split.
- **§5.5 snap-target resolution (locked 2026-05-16; Entry 19).** "Snap to [boundary]" resolves per rule type. **Working-hours violation** → the §5.5.3 next-working-day algorithm (next configured working day at that day's working-hours start). **Absolute-limit violation** → the absolute boundary (`absoluteEarliest`/`absoluteLatest`) **on the same calendar day the user originally picked** — *not* next-working-day (absolute limits are clock-time rules, not day-of-week rules; the user chose their target day deliberately). Consistent with PRD §5.1.3's "Earliest I'd ever send an email" framing.
- **§5.5 absolute snap may land on a non-working day — intentionally permitted in v1 (locked 2026-05-16).** Because an absolute snap keeps the user's chosen day, it can land on a non-working day (e.g. Sat 7:00 AM with Mon–Fri hours). **Deliberate**: the explicit day choice overrides the working-hours *soft* preference.
